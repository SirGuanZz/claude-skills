# pre-pr-review: 提交 PR 前综合代码审查

职责:在提交 PR 前对当前改动做一轮综合审查,重点捕获低级但高成本的问题:

- 安全性:注入、XSS、敏感信息泄漏、鉴权/权限绕过、不安全跳转、不安全依赖使用。
- 性能瓶颈:首屏包体积、同步重库、请求瀑布、长列表、内存泄漏、重复渲染、缓存误用。
- 类型定义完整性:接口类型、Props/Emits、API 响应、枚举/联合类型、any/断言滥用。
- 最佳实践:框架约定、状态管理、错误处理、可维护性、可访问性、测试与回归验证。

核心纪律:**只读不改代码。** 用户明确说「按审查结果修」「帮我改」后,才进入修改流程。

---

## 启动:确定审查范围

默认范围是当前未提交改动:

1. `git diff --cached`
2. `git diff`
3. `git status --short`

如果 staged 和 unstaged 都为空,告知没有可审查 diff,让用户指定:

| 范围 | 读取方式 |
|------|----------|
| 当前分支相对 main | `git diff main...HEAD` |
| 指定 commit / ref | `git show <ref>` 或 `git diff <base>..<head>` |
| 指定文件 / 目录 | 读取目标文件并结合引用搜索 |
| PR URL / 编号 | 用 `gh pr view` / `gh pr diff` 读取 |

范围明确后,一句话说明:审查模式、文件数、主要改动类型。

---

## 先读项目上下文

不要只看 diff。先用最少文件建立判断依据:

- `package.json`:框架、脚本、测试、lint、类型检查、依赖风险。
- 框架配置:`nuxt.config.*` / `vite.config.*` / `vue.config.*` / `next.config.*` / `tsconfig.json`。
- 路由入口:`pages/`、`router/`、`pages.json`、`middleware/`。
- 请求封装:`api/request.*`、`services/**`、`utils/request.*`、拦截器。
- 状态管理:`stores/**`、`store/**`、`composables/**`。
- 全局样式和设计 token:`tailwind.config.*`、`assets/styles/**`、`src/config/design.ts`。

只深入与 diff 有关的上下文。大型仓库中,优先读改动文件和它们的直接依赖/引用方。

---

## 审查维度

### 1. 安全性

必须检查:

- 用户输入是否进入 `v-html`、`innerHTML`、`dangerouslySetInnerHTML`、模板字符串 HTML、Markdown 渲染。
- URL、query、redirect、returnUrl、next 参数是否可能造成开放重定向。
- API 请求是否绕过统一 request 封装、鉴权 header、错误码处理或 CSRF 防护。
- 是否把 token、cookie、手机号、身份证、密钥、内部域名等敏感信息打到日志、URL、localStorage 或前端 bundle。
- SSR / 服务端代码是否把私密 runtimeConfig 暴露到客户端。
- 下载、预览、跳转、iframe、window.open 是否缺少协议白名单或 `noopener`。
- 依赖或动态 import 是否从用户可控路径加载。

不做攻击脚本或利用链演示;只指出风险、位置和安全改法。

### 2. 性能瓶颈

必须检查:

- 首屏同步引入重库:图表、地图、编辑器、富文本、全量 UI 库、全量 icon、moment 等。
- 顶层串行请求、重复请求、无依赖请求未并行、SSR/CSR 重复拉取。
- 大列表未分页/虚拟滚动,computed/filter/sort 在渲染路径 O(n²) 或频繁重算。
- 事件监听、定时器、IntersectionObserver、WebSocket、订阅未清理。
- 图片缺尺寸、懒加载、压缩、现代格式;首屏关键图缺 preload/priority。
- 组件无意义深响应式大对象,可用 `shallowRef` / `markRaw` / 局部状态替代。
- 动画导致 layout thrash,滚动监听未节流。

只报和本次 diff 直接相关的问题,不泛泛建议 CDN 或“可优化缓存”。

### 3. 类型定义完整性

必须检查:

- 新增/修改 API 是否有请求参数和响应类型,是否与后端字段命名一致。
- Vue Props/Emits/Slots、React Props、组件暴露方法是否完整类型化。
- store/composable 返回值是否有清晰类型边界。
- `any`、`as any`、宽泛 `Record<string, any>`、危险非空断言 `!` 是否掩盖真实分支。
- 枚举状态是否用联合类型/枚举覆盖,是否处理未知状态。
- 可空字段是否在 UI、提交、计算属性中处理。
- 表单 model、校验规则、提交 payload 类型是否一致。

允许第三方库类型缺失的最小例外,但要说明原因和收敛边界。

### 4. 最佳实践与框架约定

按项目实际技术栈检查:

- Vue/Nuxt:响应式陷阱、SSR 反模式、`useFetch`/`useAsyncData` 用法、`useState` 跨请求隔离、组件自动导入、路由跳转。
- uni-app/小程序:`pages.json`、tabBar、分包、rpx、生命周期、平台 API 调用边界。
- React/Next:Server/Client Component 边界、hooks 规则、缓存/重验证、路由跳转和 hydration。
- 通用:错误态/空态/loading、表单防重复提交、可访问性、语义标签、键盘可达、日志质量。
- 项目内约定:统一 request、路由常量、设计 token、Tailwind/UnoCSS 使用、目录命名、测试风格。

### 5. 测试与回归

必须检查:

- 新增业务分支是否有对应单测/组件测试/e2e 或手工验证清单。
- 现有测试是否需要更新快照、mock、fixture、类型断言。
- 高风险改动是否需要补类型检查、lint、构建或关键路径验证。
- UI 改动是否需要 375 / 768 / 1280 响应式检查。

没有测试框架时,输出手工验证清单,不要擅自引入依赖。

---

## 分级标准

| 等级 | 含义 | 必须处理 |
|------|------|----------|
| P0 阻断 | 安全漏洞、数据错乱、构建失败、核心路径必崩、权限绕过 | PR 前必须修 |
| P1 必改 | 明确线上风险、SSR/路由/接口错误、明显性能退化、类型掩盖真实 bug | PR 前建议修 |
| P2 建议 | 可维护性、局部性能、测试缺口、最佳实践偏离 | 可本 PR 修或记录 |
| P3 观察 | 主观风格或低收益优化 | 默认不输出,除非用户要深度审查 |

默认只输出 P0/P1/P2。不要为了凑数输出 P3。

---

## 执行流程

```text
1. 获取 diff 与文件清单
2. 读项目上下文和直接依赖/引用方
3. 按安全、性能、类型、最佳实践、测试五类审查
4. 对每条发现做证据核对:文件:行、触发条件、影响范围、建议修法
5. 内部自省一次:有没有被修复建议引出的连锁风险或遗漏的 P0/P1
6. 输出一轮报告并收口
```

大型 diff 策略:

- 改动文件 >30 个:先按风险分桶,优先审入口、request、store、公共组件、路由、配置。
- 只做抽样时必须明说抽样范围和未覆盖文件。
- 不确定的点标为「需确认」,不要脑补。

---

## 输出格式

```md
## pre-pr-review 完成

**范围**: 当前 diff / 指定 ref / 指定文件,共 N 个文件
**结论**: 可提交 / 建议修完再提交 / 阻断提交
**统计**: P0 x 个 / P1 x 个 / P2 x 个

## P0 阻断
1. `path/to/file.ts:42` — 问题标题
   - 类型:安全 / 性能 / 类型 / 最佳实践 / 测试
   - 影响:会导致什么线上问题
   - 证据:为什么确定是问题
   - 建议:具体改法或代码方向

## P1 必改
...

## P2 建议
...

## 关联影响
- `组件/模块`:引用方和可能受影响功能

## 验证清单
- [ ] typecheck / lint / test / build
- [ ] 关键业务路径
- [ ] 错误态 / 空态 / loading
- [ ] 响应式 375 / 768 / 1280(UI 改动时)

## 未覆盖 / 需确认
- 如果没有,写「无」。
```

无问题时:

```md
## pre-pr-review 完成

**范围**: 当前 diff,共 N 个文件
**结论**: 未发现 P0/P1/P2 问题,可提交。
**验证建议**: typecheck、lint、相关测试、关键页面手工验证。
```

---

## 禁止

- 禁止在审查阶段改代码。
- 禁止跳过项目上下文只凭 diff 猜测。
- 禁止泛泛输出「注意安全」「注意性能」,必须有文件、行号和可执行建议。
- 禁止把历史旧问题算到本次 PR,除非本次 diff 触发或放大该问题。
- 禁止输出 exploit 细节、攻击 payload 或绕过检测方法。
- 禁止为了显得严格而凑问题。
- 禁止在报告 / 中文回复中使用斜体(`*文字*` / `_文字_`);需要强调统一用粗体,引用文件 / 行号 / 字段名 / 命令用反引号。
